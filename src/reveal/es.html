Buildings Detection

Finding buildings among a set of segments

▼

▼

▼

Capacities developed

  • Scientific methods
  • Algorithm conception and qualification

Objective

Detect buildings on former paper printed paged maps

This is required to determine the proportion of building having electric access points inside their walls.

Result

Detects buildings with 96% accuracy

Detail

The following slides detail how this method works

Discovery

As you can see, all the buildings we have to detect are identified by a more or less well defined hatching and surrounded by the segments representing theirs walls.

The first goal is to detect this pattern

⇒ Looking for existing algorithms and scientific papers that could provide indication.

DBSCAN

Density-Based Spatial Clustering of Applications with Noise is an algorithm to detect clusters of points based on their density.

It is possible to adapt this algorithm to take into account the distance between segments and their orientation

Result

Parallel and close lines are detected correctly but artifacts are still present: buildings are by far not the only geometric shape with parallel lines.

Filtering

The previous image already had segments filtered out based on their length, parallelism, regularity, ...

Here are example of segments that were filtered out: roads' edges and text

OIGBHT Example

It transforms parallel lines into points sets. A hatching with a given angle will be represented by a vertical set of points at that angle with regularly spaced points.

Adapted OIGBHT

This formula can be adapted to suit the need we have here: finding sets of almost (angle difference of $\alpha$) parallel segments

$$OIGBHT(\theta_1, \rho_1, \theta_2, \rho_2) = \begin{cases} (\theta_1, |\rho_2-\rho_1|)& \text{if } |\theta_2 - \theta_1| < \alpha \\ undef & \text{otherwise} \end{cases}$$

OIGBHT's result

The result of this second formula shows us the sets of aligned segments.

OIGBHT's usage

This second way to detect segments is used as a filter of the DBSCAN algorithm's output. Resulting in this nicely cleaned sets of hatching.

Edges detection

Now we need to detect the edges of the buildings. Methods used:

  1. Get the edges located near the hatching and try to make a building from it
  2. Subdivide / combine / fill existing segments to create the edges of a building
  3. Genetic algorithm
  4. Alpha hull
  5. Ant's colony-based algorithm
  6. Graph (math) based solution

Basics

The basic method outputs goods results but works only on 30% of the cases.

Genetic algorithm

The genetic methods work relatively well for buildings with simple shapes

Alpha Hull

For every case that is too complicated for other algorithms: build a shape around the detected hatching

Others

The other algorithms were eliminated for the following reasons:

  • Graph-based solution: too time consuming
  • Ants colony based algorithm: unstable results

False Positive

Typical false positive case: parallel lines everywhere deceives the hatching detection algorithm.

Results

Electric Meters positioning

GPS localization of electric meters

▼

▼

▼

Capacities developed

  • Google maps API programming
  • Javascript development
  • GPS system general knowledge
  • Java SE 6
  • Swing
  • More in-depth knowledge of geodesic systems

Objectives

Find the positions of the electric meters on that path in order to enrich the SI database with geo-localized items.

Will allow to localize hard-to-find counters (in forest, with odd numerotation, ambiguous addresses, ...) for the next reading campaign.

Prototyping

Path of a technician in charge of the electric meter's reading.

Steps

  1. Synchronize the date of the meter's reading with the time of the GPS system
  2. Check the positions with a point that has no ambiguity
  3. Extract the electric meter's reading dates.
  4. Visualize the result on Google Maps

Result

The position's readings are good enough for further development.

Implementation 2

A second implementation was made to see how well it works in internal cartography tools. This time the process was improved to take into account the quality of the GPS measurements, the speed at which the technician was moving, a validity check ...

Java Implementation

The 2nd prototype was developed in Java 6, Swing. It uses the internal cartography system. It required units conversion (provided by the French National Greography Institute) between 2 systems to represent the earth:

  • WGS-84: the GPS system referential, Latitude/Longitude
  • NTF: the Lambert referential ellipsoid

The Lambert is a conical conform projection. More info here.
